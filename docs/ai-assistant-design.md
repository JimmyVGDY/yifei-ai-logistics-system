# 物流管理系统 AI 助手设计文档

> 本文档描述 AI 助手的整体设计和后续演进路线。当前项目已落地 Spring AI 只读助手：支持系统文档问答、MySQL 持久化会话、账号级长期记忆、日志排障、白名单业务查询和受控临时 SELECT 查询，不直接执行新增、修改、删除等写操作。具体配置和使用方式见 [Spring AI 接入说明](spring-ai.md)。

## 1. 设计目标

物流管理系统已经具备运单、客户、调度、轨迹、异常、费用、权限、日志和资源中心等后台能力。AI 助手的目标不是替代业务页面，而是在现有能力之上提供一个“自然语言入口”，帮助不同角色更快查询业务状态、定位问题、理解系统功能和辅助开发维护。

第一版目标：

- 支持系统使用问答：回答菜单入口、功能说明、常见问题和操作步骤。
- 支持只读业务查询：按当前登录用户权限查询运单、轨迹、异常、费用、客户等信息。
- 支持临时只读分析：复杂统计、关联、连表问题可生成候选 `SELECT`，后端校验后只读执行。
- 支持日志排障：根据 `traceId`、`operationId`、`loginSessionId`、用户编号、接口地址等条件串联操作日志和结构化日志。
- 支持业务分析：汇总待调度订单、近期异常、费用回款、客户订单走势等只读分析结果。
- 支持个性化服务：按账号长期记忆常用查询、回答风格和排障偏好，并允许用户查看、关闭、删除和清空。
- 支持开发者辅助：说明后端分层、Mapper XML、前端权限、接口约定、部署脚本和中间件链路。

明确不做：

- 第一阶段不让 AI 直接新增、修改、删除业务数据。
- 不允许 AI 执行任意 SQL；临时 SQL 只允许单条白名单 `SELECT`。
- 不让前端直接调用大模型接口，避免模型密钥和内部提示词暴露。
- 不绕过 Sa-Token 权限、菜单权限、按钮权限和用户特殊权限。
- 不把明文密码、token、完整手机号、身份证号、地址等敏感数据发送给模型。

## 2. 用户场景

| 用户角色 | AI 助手能力 | 示例问题 |
| --- | --- | --- |
| 系统管理员 | 系统状态、权限排查、日志审计、全局业务查询 | “为什么这个账号看不到费用结算？” |
| 客服人员 | 客户、订单、轨迹、异常的只读查询 | “帮我查一下客户张三最近的异常订单。” |
| 调度人员 | 待调度订单、车辆、司机、运输任务分析 | “今天还有哪些待调度订单？” |
| 司机 | 自己的运输任务、轨迹、异常上报说明 | “我当前有哪些运输任务？” |
| 财务人员 | 费用结算、收款状态、收入趋势说明 | “本月未收款费用有哪些？” |
| 客户账号 | 自己名下订单、物流轨迹、异常处理进度 | “我的订单现在到哪里了？” |
| 新开发者 | 项目结构、启动方式、接口、数据库和中间件说明 | “订单创建后 RabbitMQ 做了什么？” |

## 3. 当前组件编排

AI 助手应复用当前系统已有组件，而不是另起一套权限、日志和数据访问体系。

| 现有组件 | 当前用途 | AI 助手中的用法 |
| --- | --- | --- |
| Vue3 + Element Plus | 后台页面、表格、弹窗、菜单 | 增加 AI 助手入口，可采用右侧抽屉、悬浮按钮或独立页面 |
| `auth-store.js` | 保存登录用户、角色、权限、菜单 | 提供当前用户身份和权限给 AI 页面做按钮/入口显隐 |
| Sa-Token | 登录认证、会话、权限码校验 | 后端 AI 接口统一校验登录态和权限码 |
| 角色权限 + 用户特殊权限 | 控制菜单、接口、按钮 | AI 工具调用前二次校验，保证只能访问当前用户可见数据 |
| MyBatis Mapper XML | 业务 SQL 集中维护 | 标准 AI 查询复用 Service/Mapper；复杂临时分析走受控只读 SQL 网关，普通业务 SQL 仍进入 XML |
| Elasticsearch | 订单搜索 | 第一阶段复用订单搜索，后续扩展文档知识库检索 |
| Redis | 订单缓存、上下文数据 | 缓存最近 AI 上下文、热点问答和限流辅助数据；会话历史以 MySQL 为准 |
| Qdrant | AI 长期记忆向量召回 | 按 `userId/userCode` 召回账号级长期偏好和使用习惯 |
| RabbitMQ | 订单事件异步处理 | 长耗时 AI 分析任务可异步化，避免阻塞前端请求 |
| Sentinel | 接口限流、降级保护 | 保护 AI 问答接口和高频查询工具 |
| 操作日志表 | 记录关键用户操作 | 记录 AI 问答、工具调用、查询目标和执行结果 |
| 结构化日志 | 文件级 JSON 日志检索 | 为日志排障助手提供 trace 链路数据 |
| `traceId` | 单次请求/业务链路追踪 | 串联一次 AI 问答内的模型调用、工具查询和业务日志 |
| `operationId` | 单次可审计操作 ID | 标识一次 AI 问答或一次 AI 工具调用 |
| `loginSessionId` | 登录会话审计标识 | 查询某个用户一次登录期间所有 AI 行为 |

## 4. 当前落地状态

当前已实现内容：

- 后端新增 `jimmy.ai` 模块，统一通过 `/ai/**` 接口提供能力。
- 接入 Spring AI `ChatClient`，模型密钥缺失时自动降级为本地兜底回答。
- 通过 `README.md` 和 `docs/*.md` 做轻量文档检索。
- 复用通用模块查询、看板查询和权限数据范围，支持普通只读业务查询。
- 对统计、关联、连表类问题提供临时只读 SQL 网关，执行前做只读、白名单、权限和脱敏校验。
- 复用操作日志查询能力，按 `traceId`、`operationId`、`loginSessionId`、接口地址、用户编号和时间范围做日志排障。
- MySQL 保存当前用户 AI 会话和消息明细，支持归档、恢复、逻辑删除和服务重启后回显。
- Redis 缓存最近上下文和热点问答，缓存失效时从 MySQL 回填，不再作为会话历史唯一来源。
- MySQL + Qdrant 保存账号级长期记忆，支持召回、自动写入、删除、清空和关闭。
- Sa-Token 控制 `ai:chat`、`ai:log:analyze`、`ai:conversation:query`、`ai:conversation:archive`、`ai:conversation:delete`、`ai:memory:query`、`ai:memory:delete`、`ai:memory:settings`。
- 前端新增 `/ai-assistant` 页面，包含会话列表、长期记忆设置、问答区、引用来源和日志排障时间线。

当前未实现内容：

- 未开放写操作。
- 未开放客户账号的临时关联 SQL，客户数据仍通过普通查询路径隔离。
- 未做长任务异步分析。

## 5. 总体架构

建议架构如下：

```text
Vue3 AI 助手入口
        |
        v
AI Assistant Controller
        |
        v
AI Assistant Service
        |
        +--> 权限上下文：Sa-Token / 角色权限 / 用户特殊权限
        +--> 工具编排：模块查询 / 看板查询 / 临时只读 SQL / 日志查询
        +--> 知识检索：项目文档 / 接口文档 / 数据库说明
        +--> 长期记忆：MySQL 画像 / Qdrant 向量召回 / 记忆事件审计
        +--> 审计记录：操作日志 / 结构化日志 / traceId / operationId / loginSessionId
        |
        v
模型网关或模型客户端
        |
        +--> 云端大模型
        +--> 本地模型
        +--> 企业内网模型
```

设计原则：

- 前端只调用系统后端，不直接调用模型平台。
- 后端统一做鉴权、脱敏、限流、审计和模型参数控制。
- AI 不直接绕过后端访问数据库；标准查询必须通过已有 Service，临时 SQL 必须通过专用只读网关和安全校验器。
- 所有工具调用都有明确权限码和操作日志。
- 所有模型输入输出都要经过敏感信息过滤。

## 6. 前端设计

前端可采用三种入口形态：

| 入口形态 | 优点 | 适用阶段 |
| --- | --- | --- |
| 右侧抽屉 | 不打断当前页面，可结合当前页面上下文提问 | 推荐第一阶段 |
| 悬浮助手按钮 | 任何页面都可唤起，体验轻量 | 推荐第一阶段 |
| 独立 AI 助手页面 | 适合复杂日志检索、长报告、开发问答 | 第二阶段 |

当前第一版已经采用独立页面 `/ai-assistant`，并保留后续扩展为右侧抽屉或悬浮助手的空间。

第一阶段已实现：

- 按 `ai:chat` 权限控制入口和路由访问。
- 打开后展示会话列表、输入框、回答区域、引用来源、工具调用结果和日志分析面板。
- 根据 `auth-store.js` 中的权限决定入口是否可见。
- 回答中展示引用来源，例如“来自操作日志”“来自订单表”“来自系统文档”。
- 对可能涉及写操作的问题，只返回建议和跳转入口，不直接提交。

前端展示建议：

- 普通问答：展示自然语言回答。
- 业务查询：展示表格摘要，并提供“跳转到对应页面”的链接。
- 日志排障：展示链路时间线、接口、状态、耗时、异常摘要。
- 权限排查：展示菜单权限、接口权限、按钮权限和用户特殊授权差异。

## 7. 后端设计

当前后端已经新增 AI 助手相关分层：

```text
controller
└── AiAssistantController

service
├── AiAssistantService
├── AiKnowledgeService
├── AiBusinessQueryTools
├── AiToolCallContext
├── AiReadonlyQueryService
├── AiGeneratedSqlQueryService
├── AiSqlSafetyValidator
├── AiLogAnalysisService
├── AiConversationService
├── AiMemoryService
├── AiQdrantMemoryClient
├── AiMemoryVectorEncoder
├── AiSensitiveDataMasker
└── AiModelGateway

model
├── AiChatRequest
├── AiChatResponse
├── AiToolCall
├── AiCitation
├── AiConversationVO
├── AiMemoryProfileVO
└── AiMemoryItemVO
```

职责划分：

- `AiAssistantController`：接收前端问答请求，返回统一 `ApiResponse<T>`。
- `AiAssistantService`：负责编排文档检索、Spring AI Tool Calling、规则兜底、审计和模型回答。
- `AiKnowledgeService`：检索文档、接口说明、数据库说明和后续知识库。
- `AiBusinessQueryTools`：把单模块查询、全场景模糊搜索、业务联合查询、看板查询和临时只读 SQL 包装成 Spring AI 只读工具。
- `AiToolCallContext`：收集模型工具调用产生的引用来源、工具调用摘要和业务上下文，供前端展示和操作日志审计。
- `AiReadonlyQueryService`：复用通用模块、看板和数据范围查询能力，为 AI 执行单模块、全场景模糊、自动联合查询并生成只读业务摘要。
- `AiGeneratedSqlQueryService`：处理统计、关联、连表等临时分析问题，只执行通过校验的候选 `SELECT`。
- `AiSqlSafetyValidator`：校验候选 SQL 的只读语义、单语句、表字段白名单、敏感字段和当前用户权限。
- `AiQueryIntentParser`：先归一化用户输入，再识别白名单模块、关键词、状态、车牌号、业务编号和时间范围。
- `AiConversationService`：以 MySQL 保存当前用户会话和消息明细，Redis 只缓存最近上下文，并为“只要待处理的”这类追问提供上一轮查询上下文。
- `AiMemoryService`：管理账号级长期记忆画像、Qdrant 召回、自动写入、删除、清空和审计事件。
- `AiQdrantMemoryClient`：封装 Qdrant 集合创建、维度校验、向量点写入、搜索和删除；不可用或维度不匹配时降级。
- `AiMemoryVectorEncoder`：优先使用 Ollama `bge-m3` 生成 1024 维向量，不可用时降级为确定性哈希向量并补齐到 1024 维。
- `AiLogAnalysisService`：复用操作日志生成排障摘要、风险点、建议和时间线。
- `AiSensitiveDataMasker`：模型输入前清洗敏感字段。
- `AiModelGateway`：屏蔽不同模型供应商差异，统一超时、重试、脱敏和错误处理。

## 8. 知识库与长期记忆来源

系统文档知识库当前仍以“结构化文档索引 + 关键字检索 + 业务工具”为主；账号级长期记忆已经接入 Qdrant，用于召回用户偏好和使用习惯。

| 来源 | 内容 | 处理方式 |
| --- | --- | --- |
| `README.md` | 项目简介、快速启动 | 文档索引 |
| `docs/*.md` | 架构、接口、权限、日志、数据库、开发规范 | 文档索引 |
| `schema.sql` / 增量脚本 | 表结构和字段含义 | 生成数据库说明索引 |
| Controller/接口文档 | 接口路径、参数、权限 | 生成接口索引 |
| 操作日志表 | 用户操作、请求地址、耗时、异常 | 只读查询工具 |
| 结构化日志文件 | trace 链路、系统异常、SQL/中间件日志 | 日志检索工具 |
| 业务表 | 运单、客户、轨迹、异常、费用 | 通过 Service 做权限过滤后查询 |
| 受控临时 SQL | 统计、关联、连表类只读分析 | 由模型生成候选 `SELECT`，模型自检后再由后端白名单、权限、安全校验和 `limit 0` 语法预检；语法失败时最多自动纠错 3 次 |
| `ai_user_profile` | 当前账号记忆开关、回答风格、常用模块和查询习惯 | MySQL 真值，可审计、可关闭 |
| `ai_user_memory` | 长期记忆分类、置信度、脱敏摘要和 Qdrant 向量 ID | MySQL 真值，可删除、可清空 |
| Qdrant | 脱敏记忆摘要向量点 | 按 `userId/userCode` 过滤召回，不作为唯一真值 |

当前业务数据查询已经升级为“Spring AI Tool Calling + 后端安全工具层”：

- 用户明确模块时，调用单模块只读查询工具。
- 用户只输入短词或模糊问题时，调用全场景模糊搜索工具，在当前账号有权限的模块中召回。
- 用户提出客户全貌、订单完整链路、司机任务链路、车辆任务链路、异常影响等问题时，调用业务联合查询工具。
- 用户提出统计、排名、汇总、关联或连表分析时，才调用临时只读 SQL 工具。
- 如果模型不可用或没有主动调用工具，后端继续使用 `AiQueryIntentParser` 做规则兜底。

长期记忆写入规则：

- 自动记忆默认开启，但用户可以关闭、删除单条或清空全部。
- 记忆提炼优先使用 LLM 判断：每次 AI 回答完成后，由一个轻量分类提示词判断对话是否暴露了用户偏好、习惯或格式要求。LLM 判断"不值得记忆"时不会写入任何内容，从根本上避免"查什么存什么"。
- LLM 不可用时自动降级为关键词匹配：仅抓取"以后都这样""我主要查""先给结论"等明确表达偏好的语句，不再使用兜底规则。
- 普通的一次性业务查询（如"查订单123""最近三天的新单子"）不会触发任何记忆写入。
- 只记录脱敏后的回答风格偏好、常用模块、常用筛选条件、常查业务对象和排障偏好。
- 密码、token、密钥、完整手机号、完整地址、SQL 原文和异常堆栈不得写入 MySQL 记忆表或 Qdrant payload。
- Qdrant 不可用时不影响 AI 问答和物流业务，只记录降级审计事件。

后续如果文档量变大，可以引入文档向量检索：

- 将项目文档切片后写入 Elasticsearch 或独立向量库。
- 每个文档片段记录来源文件、标题、更新时间和版本。
- 回答时必须返回引用来源，方便开发者核对。

## 9. 权限与数据隔离

AI 助手必须遵守“用户在页面上看不到的数据，AI 也不能回答”的原则。

权限控制建议：

- AI 入口权限：通过 `ai:chat` 控制是否能使用 AI 助手；该权限只展开当前用户自己的会话和长期记忆能力，不自动展开日志分析。
- 日志分析权限：`ai:log:analyze` 属于跨用户审计能力，默认只应授予管理员、审计员和确需排障的管理角色。
- 工具权限：每个工具绑定现有业务权限码，例如订单查询绑定 `logistics:order:query`。
- 数据权限：客户账号只能查询关联客户的数据，司机只能查询自己的任务和轨迹。
- 特殊授权：继续复用用户额外授权和用户禁用权限，优先级与当前系统保持一致。
- 按钮权限：涉及“生成处理建议”“跳转编辑页面”等入口时，也要判断对应增删改查权限。

典型规则：

| 问题类型 | 权限判断 |
| --- | --- |
| “查某订单状态” | 需要订单查询权限，并校验数据归属 |
| “分析某用户操作日志” | 需要操作日志查询或审计权限 |
| “为什么看不到菜单” | 需要权限配置查询权限，普通用户只能查自己 |
| “帮我删除这个订单” | 第一阶段拒绝执行，只给出人工操作路径 |

临时只读 SQL 的额外规则：

- 只能生成一条 `SELECT`，禁止写入、更新、删除、建表、删表、过程调用和文件读取。
- 只能使用 `AiSqlSafetyValidator` 暴露给模型的表和字段，禁止 `select *` 和敏感字段。
- 每张表都要匹配当前账号的查询权限；权限不足时只返回用户友好的中文提示。
- 客户账号不开放临时关联 SQL，避免绕开客户数据范围，仍使用客户、订单、轨迹等普通只读查询。
- 返回结果最多 20 行，并在模型和前端展示前继续脱敏。

## 10. 日志审计与链路追踪

AI 助手相关行为必须进入现有审计体系。

记录内容：

- `traceId`：一次 AI 请求链路。
- `operationId`：一次 AI 问答或工具调用。
- `loginSessionId`：当前登录会话。
- `userId`、`userCode`、`roleCode`：原值保留，便于审计。
- `usernameMasked`：用户名脱敏展示。
- `module=ai-assistant`。
- `operation`：例如“AI助手-普通问答”“AI助手-日志分析”“AI助手-订单查询”。
- `requestUri`、`method`、`costMs`、`result`、`errorMessage`。
- 工具调用摘要：工具名、目标对象、权限码、返回数量。

脱敏要求：

- 输入给模型前脱敏手机号、身份证号、邮箱、详细地址、token、密码。
- 操作日志中保留用户 ID、业务 ID、traceId、operationId 等追踪标识原值。
- 展示给前端的日志详情默认折叠长参数，避免操作日志页面过宽。
- 敏感字段只展示摘要，不展示完整原文。

## 11. 问答流程

普通问答流程：

```text
用户输入问题
  -> 前端携带 token 请求 /ai/chat
  -> 后端校验登录态和 ai:chat 权限
  -> 生成 traceId/operationId
  -> 判断问题类型
  -> 检索系统文档、调用普通只读工具或临时只读 SQL 网关
  -> 脱敏后发送模型
  -> 生成回答和引用来源
  -> 写入操作日志
  -> 返回前端展示
```

业务查询流程：

```text
用户询问业务数据
  -> 识别查询意图和实体
  -> 全量词只确定模块，纯名称默认作为客户/联系人线索
  -> 筛选式追问才继承上一轮模块，确认式补充复用上一轮关键词
  -> 检查对应业务权限码
  -> 通过 Service 查询数据；复杂统计类问题可进入临时只读 SQL 网关
  -> 应用角色/用户数据范围
  -> 汇总为自然语言 + 表格摘要
  -> 返回引用来源和跳转链接
```

日志排障流程：

```text
用户输入 traceId / operationId / loginSessionId
  -> 校验日志查询权限
  -> 查询操作日志表
  -> 查询结构化日志文件或日志索引
  -> 按时间排序生成链路时间线
  -> 标记异常节点、慢接口、权限失败和中间件失败
  -> 返回排障建议
```

## 12. 模型接入方式

| 方案 | 优点 | 风险 | 建议 |
| --- | --- | --- | --- |
| 云端大模型 | 能力强、接入快、维护成本低 | 数据出网、成本和合规要求高 | 适合个人学习和低敏环境 |
| 本地模型 | 数据不出本机，成本可控 | 部署复杂、效果依赖硬件 | 适合离线演示和内网环境 |
| 企业内网模型平台 | 合规性好，可统一审计 | 依赖企业平台能力 | 适合正式企业部署 |

无论选择哪种模型，都建议通过后端 `AiModelGateway` 统一封装：

- 模型供应商配置走后端配置文件或环境变量。
- API Key 不进入前端代码和 Git 仓库。
- 设置超时时间、最大 token、重试次数和降级文案。
- 统一记录调用耗时、失败原因和模型名称。

## 13. 风险控制

| 风险 | 处理策略 |
| --- | --- |
| 越权查询 | 每个工具调用前做 Sa-Token 权限和数据范围校验 |
| 敏感数据泄露 | 模型输入前统一脱敏，日志展示默认摘要 |
| 提示词注入 | 系统提示词和工具权限由后端控制，不信任用户输入中的“忽略规则”等指令 |
| 幻觉回答 | 对业务数据和系统文档回答必须返回引用来源 |
| 高成本调用 | Redis 缓存热点问答，Sentinel 限流，设置每日/每用户额度 |
| 写操作误执行 | 第一阶段不开放写操作；后续必须二次确认并记录审计 |
| AI 生成危险 SQL | 候选 SQL 只作为文本，后端强制校验单条 SELECT、白名单表字段、权限和敏感字段 |
| 用户误输符号或空格 | 意图解析前先做全角、零宽字符、重复空格和边界标点归一化 |
| 上下文丢失 | 同一 Redis 会话内只对筛选式追问继承上一轮模块，纯名称输入默认按客户线索查询 |
| 纯名称误判模块 | “陈土豆”这类无模块输入先查客户/联系人；“是一个客户”这类确认式补充复用上一轮关键词 |
| 全量查询误生成关键词 | “所有运单和订单”“全部客户管理数据”只识别模块，不把范围词和模块名当 keyword |
| 订单和运单中心混淆 | 订单管理用于业务订单和下单记录；运单中心用于运输单、WB 运单号和运输状态 |
| 单模块无结果 | 客户模块未命中时，用同一关键词在当前账号可访问的订单、运单、任务、轨迹、异常和费用中全局只读召回 |
| 向量召回误命中 | 向量库只能用于候选模块和文档语义召回，最终业务数据必须走白名单查询和权限校验 |
| 长任务阻塞 | 报表分析、日志大范围扫描等使用 RabbitMQ 异步处理 |
| 日志过宽 | 前端摘要展示，详情抽屉展开，长 JSON 支持折叠 |

## 14. 分阶段落地路线

### 第一阶段：只读问答

- 已新增 AI 助手入口和 `/ai/chat`。
- 已接入项目 Markdown 文档轻量检索。
- 已支持系统功能问答和开发者问答。
- 已支持引用来源展示。

### 第二阶段：业务查询

- 已增加标准业务只读查询，复用现有模块分页、看板、权限和客户数据范围。
- 已增加临时只读 SQL 网关，用于复杂统计、关联和连表分析。
- 回答中返回摘要表格、引用来源和工具调用摘要。

### 第三阶段：日志排障

- 已接入操作日志表；结构化日志文件/日志索引为后续增强。
- 支持按 `traceId`、`operationId`、`loginSessionId` 分析链路。
- 输出时间线、失败节点、慢接口和处理建议。

### 第四阶段：确认式操作

- 支持“生成操作草稿”，例如异常处理意见、客户备注、费用催收建议。
- 用户点击确认后才调用业务接口。
- 每次确认操作都记录原始建议、确认人、目标对象和执行结果。

## 15. 后续任务清单

后续真正开发时建议按以下任务拆分：

- 增强文档索引构建脚本。
- 继续扩展只读工具白名单和权限映射。
- 新增 AI 操作日志分类和前端查询过滤项。
- 评估是否补充右侧抽屉或悬浮入口。
- 新增日志排障时间线组件。
- 增加 AI 问答脱敏测试、权限测试和日志审计测试。

## 16. 当前结论

在当前系统中接入 AI 助手是可行的，而且和现有技术栈契合度较高。当前路线是把 AI 助手做成只读的“系统问答 + 业务查询 + 临时统计分析 + 日志排障”入口，复用已有 Sa-Token、MyBatis、Redis、RabbitMQ、Elasticsearch、Sentinel、操作日志和结构化日志能力。临时 SQL 只作为受控只读分析网关存在，普通业务 SQL 仍遵守 Mapper XML 规范。等权限、审计、脱敏和引用来源都稳定后，再考虑带二次确认的业务操作能力。
